# 开发流程章程 (Development Workflow)

> 本文档定义了本项目的标准开发流程。所有工作必须按此流程执行。

## 核心原则

1. **文件即状态** - 我不依赖记忆，只读取文件判断进度
2. **文档先行** - 代码变更前必须有文档变更
3. **测试门控** - 测试不通过不允许进入下一阶段
4. **自动提交** - 合适的时机自动提交Git，无需等待确认

## 阶段定义（文件状态驱动）

```
需求文档 ──→ 实现计划 ──→ 代码实现 ──→ 测试通过 ──→ 发布归档
   ↑                                          │
   └────────── 失败时回滚到上一阶段文件 ────────┘
```

| 阶段 | 检查文件 | 完成标准 |
|------|----------|----------|
| **P1 需求** | `docs/specs/*` | 文档已更新，描述清晰 |
| **P2 计划** | `docs/plans/impl-plan.md` 任务条目 | 任务已分解，有验收标准 |
| **P2.5 活跃** | `docs/process/active-tasks.md` | 任务已移入活跃看板 |
| **P3 实现** | 代码文件 + `tests/` 测试 | 单元测试通过 |
| **P4 验证** | 测试全部通过 | `make test` 或 `npm test` 0 failures |
| **P5 发布** | `CHANGELOG.md` + 版本号更新 | PyPI/GitHub 同步完成 |

## 我的执行规则

### 规则 1：进入项目先读状态

每次进入项目目录，我必须按顺序读取：
1. `CLAUDE.md` - 项目上下文
2. **`docs/process/workflow.md`** - **必读：开发流程章程**
3. **`docs/process/active-tasks.md`** - 当前进行中的任务
4. `docs/plans/impl-plan.md` - 待规划任务
5. `CHANGELOG.md` - 最新版本（如果需要发布）

### 规则 2：文档变更检查

**任何代码修改前**，我必须检查对应的文档是否已更新：

| 变更类型 | 必须先更新的文档 |
|----------|------------------|
| 新功能 | `docs/plans/impl-plan.md` 任务定义 → `docs/process/active-tasks.md` |
| Bug修复 | `CHANGELOG.md` 记录 + `docs/process/active-tasks.md` |
| 配置变更 | 相关 spec/design 文档 |
| 接口变更 | API 文档（docstring也算） |

**任务追踪流程**：
1. 新需求 → 更新 `impl-plan.md`（分配Task编号）
2. 准备执行 → 移入 `active-tasks.md`（标记为"进行中"）
3. 开发完成 → 更新 `active-tasks.md`（标记为"待验证"）
4. 验证通过 → 更新 `CHANGELOG.md` → 从 `active-tasks.md` 移除

**如果文档未更新**：
- 小改动（<10行）：我可以边改代码边补文档，但必须同一次提交
- 大改动：必须暂停，先写文档

### 规则 3：测试强制执行

**每个代码变更必须有测试覆盖**，按以下优先级：

| 覆盖级别 | 要求 | 适用场景 |
|----------|------|----------|
| **核心功能** | 100% 覆盖 | 所有P0功能、安全相关代码 |
| **受影响功能** | 100% 覆盖 | 本次改动可能影响的其他模块 |
| **直接功能** | 100% 覆盖 | 本次改动直接涉及的新功能/修复 |
| **周边功能** | 回归测试通过即可 | 与改动无直接关联的模块 |

**测试执行顺序**：
1. 先跑改动模块的单元测试（快速反馈）
2. 再跑受影响模块的测试（识别副作用）
3. 最后全量测试（最终验证）

**测试失败处理**：
1. 分析失败原因
2. 修复代码（首选）或修正测试（如果测试本身有问题）
3. 重新运行测试
4. **测试通过前不允许提交到主线**

### 规则 4：提交时机

**以下情况必须立即提交**：
- 单个任务完成（功能/修复/测试通过）
- 文档更新完成
- 版本号更新完成
- 每天工作结束前（无论是否完成）

**提交信息格式**：
```
[Task X] 简要描述

- 变更点1
- 变更点2

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```

### 规则 5：失败回滚

如果某个阶段失败且无法快速修复（>15分钟）：
1. 记录失败原因到注释或文档
2. **回滚到上一阶段的稳定状态**（git reset/stash）
3. 重新分析并修改文档
4. 重新开始实现

### 规则 6：发布策略

**不自动发布，累积后主动询问**：

1. **单个任务完成后**：提交到GitHub，不发布
2. **一批任务完成后**（由你指定的多个任务完成）：主动询问是否发布
3. **发布前检查清单**（必须全部完成）：
   - [ ] CHANGELOG.md 已更新（记录本次发布的所有变更）
   - [ ] 版本号已更新（pyproject.toml, cli.py, package.json）
   - [ ] 全量测试通过
   - [ ] 文档已同步（README, DEPLOYMENT, CLAUDE等）
   - [ ] GitHub已push（提交 + tag）
   - [ ] PyPI已上传（build + twine upload）
   - [ ] 安装验证通过（`pipx install peekview` 测试）

**版本号规则**：
- `MAJOR`: 不兼容变更（MVP阶段不考虑）
- `MINOR`: 新功能
- `PATCH`: Bug修复

## 我的限制（我必须承认的）

### 我无法做到：
1. ❌ 跨会话记住之前的决定
2. ❌ 主动发现"需要做什么"（除非你告诉我或写在文件里）
3. ❌ 长时间运行任务（>30分钟）
4. ❌ 与外部系统交互（CI/CD、PR review等）

### 我必须依赖：
1. ✅ 文件作为唯一状态源
2. ✅ 你提供的明确指令
3. ✅ 自动化测试作为质量门控
4. ✅ Git历史记录变更

## 实际执行流程示例

### 场景：添加新功能 "支持Markdown导出"

**Step 1: 检查状态**
```
读 INDEX.md → 发现 Task 17: Markdown导出（待实现）
```

**Step 2: 文档检查**
```
impl-plan.md 中 Task 17 已有定义？
  ├── 否 → 更新 impl-plan.md，定义任务
  └── 是 → 继续
```

**Step 3: 实现**
```
编写代码 → 编写测试 → 运行测试
  ├── 失败 → 修复（循环最多3次）→ 仍失败 → 回滚
  └── 通过 → 继续
```

**Step 4: 更新状态**
```
更新 INDEX.md: Task 17 状态改为 "已完成"
更新 CHANGELOG.md: 添加 0.1.4 版本说明
```

**Step 5: 提交**
```
git add -A
git commit -m "[Task 17] Add Markdown export support

- Add /api/v1/entries/{slug}/export endpoint
- Support md/html format
- Add CLI export command

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>"
git push origin main
```

**Step 6: 发布（如果需要）**
```
检查是否需要发布（CHANGELOG是否有未发布内容）
  ├── 是 → 更新版本号 → 构建 → 上传PyPI → 打git tag
  └── 否 → 结束
```

## 质量保证检查点

每个会话结束前，我必须确认：

- [ ] 修改的文档已保存
- [ ] 代码变更有对应测试
- [ ] 测试全部通过
- [ ] 已提交到Git（除非明确在WIP分支）
- [ ] 版本号/CHANGELOG已同步（如果是发布）

## 例外处理

以下情况可以跳过文档先行：
1. **Typo修复** - 明显的拼写错误
2. **格式化** - 仅代码风格调整（ruff format）
3. **测试数据** - 仅修改fixtures/test data
4. **配置微调** - 调整开发配置（端口、超时等）

**但以上变更仍需测试通过才能提交。**
